# NAV 시뮬레이션 설계 문서
## 신재생에너지 펀드형 STO (태양광 PF)

---

# 1. 목적

본 문서는 태양광 프로젝트 파이낸싱(PF) 구조 기반 펀드형 STO의 월별 NAV 경로 시뮬레이션을 위한 백본 아키텍처를 정의한다.

시뮬레이션 요구사항:

- 월 단위 타임스텝 사용
- 시뮬레이션 범위:
  - 건설기간 1년 (12개월)
  - 운영기간 20년 (240개월)
  - 전체 시뮬레이션 40년 (480개월, 롤오버 포함)
- NAV 시계열 생성
- 희소 외생 포아송 이벤트 반영
- 마르코프 상태 갱신 구조 준수
- 다중 PF 운영 및 롤오버 편입 지원
- 토큰 병합(Reverse Split) 메커니즘
- 태양광 패널 열화(Degradation) 반영

본 문서는 최종 가치평가 모델이 아니다.
시뮬레이션 엔진의 백본 구조를 정의하는 문서이다.

---

# 2. 시간 구조

- 타임스텝: 1개월
- 건설기간: 12개월
- 운영기간: 240개월
- 단일 PF 수명: 252개월
- 전체 시뮬레이션: 480개월 (40년)
- 롤오버 주기: 12개월 (1년)

# 3. 자본구조

## 3.1 단일 프로젝트 자본구조

프로젝트당 자금조달 규모: 20억원

| 구분 | 금액 | 비율 |
|------|------|------|
| 자기자본 (사업 관련 투자자) | 4억원 | 20% |
| 선순위 대출 | 8억원 | 40% |
| 토큰화 대상 | 8억원 | 40% |

## 3.2 다중 PF 운영 구조

- 초기 운영 PF 수: 1개
- 롤오버 시 편입 PF 수: 1개 (동일 규모)
- 초기 자금조달 규모: 20억원 (20억 × 1개)

## 3.3 토큰 단위

소액 투자자 접근성을 위해 토큰 1개당 가격을 1만원으로 설정한다.

| 항목 | 값 |
|------|-----|
| 토큰 1개당 가격 | 10,000원 |
| PF당 발행 토큰 수 | 80,000개 (= 8억 ÷ 1만원) |
| 초기 발행 토큰 수 | 80,000개 (= 80,000 × 1개 PF) |
| 초기 토큰 자금조달 총액 | 8억원 |

# 4. 수익 분배 구조

순현금흐름 분배:

- 순현금흐름 = 지분수익 + 토큰수익
- 분배 비율: 지분수익 : 토큰수익 = 1 : 2 (자기자본 4억 : 토큰 8억 비율)
- 지분수익 비율: 1/3 (≈33.3%)
- 토큰수익 비율: 2/3 (≈66.7%)

---

# 5. 모델링 프레임워크

마르코프 상태 전이 구조를 사용한다.

매월 t에서:

State S_t → 외생 충격 → 전이 → S_{t+1}

파생 변수 (PV_t, P_complete_t, NAV_t)는 S_t로부터 계산된다.

---

# 6. 상태 변수

상태 S_t에 포함되어야 할 변수:

## 핵심 상태

- `time_t` : 정수 (현재 월 인덱스)
- `status_t` : 범주형
  - "PRE"  : 완공 전
  - "POST" : 완공 후
  - "FAILED" : 건설 디폴트
- `token_count_t` : 총 발행 토큰 수
- `monthly_distribution_t` : 월별 분배금 (펀드 전체)

## 시장 데이터 윈도우

윈도우 크기: n개월 (기본값 6개월)

- `SMP_t` : shape (n, 2)
  - [가격, 거래량]
- `REC_t` : shape (n, 2)
  - [가격, 거래량]
- `energy_amount_t` : shape (n,)

t-n+1월부터 t월까지의 데이터를 나타낸다.

## 재무 파라미터

- `r_t` : 고정 사회적 할인율
  - 연 4.5% 고정

## 외생 이벤트 플래그

- `curtail_today` : bool (당월 출력제한 발생 여부)

# 7. 외생 이벤트 (포아송 기반)

월별 베르누이 시행 (포아송 근사)을 가정한다.

## 7.1 건설 디폴트 이벤트

- status == PRE 일 때만 적용
- 월별 확률:
  p_default = 0.3% = 0.003

발생 시:
→ status = FAILED
→ NAV = 0 (영구)

---

## 7.2 운영 출력제한 이벤트

- status == POST 일 때만 적용
- 월별 확률:
  p_curtail = 0.3% = 0.003

발생 시:
→ 해당월 SMP 및 REC 수익 = 0
→ PV 계산에 반영

---

# 8. 완공 모델

status == PRE 동안 완공 확률이 매월 갱신된다.

## 8.1 완공 확률 정의

P(t) = h(t) × g(t)

- h(t): 위험률 함수 (hazard rate function)
- g(t): 바이너리 변수 (0 또는 1)

## 8.2 g(t) 확률 과정

- 매 시점 t에서 0이 될 확률: 0.3% (0.003)
- 포아송 분포가 아닌, 독립적 베르누이 시행
- g(t) = 0 발생 시 → 프로젝트 실패 (FAILED)

## 8.3 상태 전이

status == POST 인 경우:
P_t = 1

# 9. 비용구조 및 FCF

## 9.1 비용 구조

| 항목 | 연간 | 월간 | 적용 기간 |
|------|-----:|-----:|-----------|
| OPEX | 1,000만 원 | 83만 원 | 운영 전 기간 (20년) |
| 사무위탁비 | 1,000만 원 | 83만 원 | 운영 전 기간 (20년) |
| 대수선비 | 500만 원 | 42만 원 | 운영 전 기간 (20년) |
| **(고정비용 소계)** | **2,500만 원** | **208만 원** | |
| 선순위 이자 | 4,800만 원 | 400만 원 | 운영 1~15년차만 |
| 대출 원금상환 | 5,333만 원 | 444만 원 | 운영 1~15년차만 |
| **월 총비용 (1~15년)** | | **1,053만 원** | |
| **월 총비용 (16~20년)** | | **208만 원** | 80% 감소 |

## 9.2 FCF (Free Cash Flow) 정의

```
Gross CF = Revenue - OPEX - 이자 - 원금상환
세후 FCF = Gross CF × (1 - 0.22)    (Gross CF > 0일 때)
token_cf = 세후 FCF × (2/3)
```

## 9.3 PV 엔진

함수:

PV_t = pv_engine(state, mode)

프로젝트 생존 조건 하에서 미래 운영 현금흐름의 현재가치를 반환한다.

```
V = Σ [ Expected FCF_t / (1 + r)^t ]
```

- 할인율 r = 사회적 할인율 4.5%
- 열화 반영 감소 연금 (Declining Annuity) 사용

입력:
- SMP 윈도우
- REC 윈도우
- 발전량 윈도우
- 할인율
- 출력제한 플래그

출력:
- 스칼라 PV_t

내부 예측 로직은 모듈화되어 교체 가능하다.

# 10. 태양광 패널 열화 (Degradation)

연 0.5%의 열화율을 적용한다.

```
degradation_factor = (1 - 0.005) ^ (운영 경과 월수 / 12)
```

- 건설 중(PRE)에는 열화 없음 (계수 = 1.0)
- 월별 매출액에 직접 곱해져 반영
- PV 산출 시 감소 연금현가계수 사용

---

# 11. SMP/REC 모델링

## 11.1 가격 과정

SMP와 REC는 GBM(기하브라운운동) 모델을 따른다.

| 항목 | SMP | REC |
|------|-----|-----|
| 초기가격 | 113 원/kWh | 77 원/kWh |
| 드리프트 | +6%/년 | +6%/년 |
| 변동성 | 16%/년 | 14%/년 |
| 하한 | 79 원 | 40 원 |
| 상한 | 159 원 | 134 원 |

**변동성 조정 근거**: 고정가격이 6개월 평균으로 산정되므로, 이론적으로 표본평균의 표준편차 = σ/√6 ≈ 41% 수준으로 감소함. 보수적으로 원래 변동성의 절반(50%)을 적용.

## 11.2 장기계약 구조

편입 당시 T 시점에서 (SMP + 1×REC) 장기계약 체결

- 계약 가격은 편입 시점의 SMP, REC 가격 기준으로 고정

## 11.3 고정가격 산정

프로젝트 완공 시점에서 직전 6개월의 과거 SMP, REC 데이터 평균으로 산정

---

# 12. NAV 계산

NAV_t = P_t × PV_t (위험중립 평가)

- 건설 단계: NAV_t = P_t × PV_t (완공 확률 반영)
- 운영 단계: NAV_t = PV_t (P_t = 1.0)
- 실패: NAV_t = 0

---

# 13. 전이 함수

매 타임스텝에서:

1. 시장 데이터 전진 (GBM)
2. 모든 PF 업데이트 및 NAV 합산
3. 롤오버 훅 (신규 PF 편입)
4. 토큰 병합 체크
5. 펀드 전체 상태 결정

완공 시점에 추가적인 변동성 충격 없음.
인위적인 점프 추가 없음.
체제 전환(regime switch)이 자연스럽게 NAV 구조를 변경함.

# 14. 롤오버 구조 및 토큰 증자

## 14.1 롤오버 필요성

단일 프로젝트는 만기 시 가치가 0이 되는 구조.
롤오버를 통해 영속적인 인프라 투자수단으로 설계.

## 14.2 롤오버 주기

- 12개월(1년)마다 롤오버 시도
- 매 롤오버 시 1개 PF 편입 (동일 20억 규모)

## 14.3 변수 정의

| 변수 | 설명 |
|------|------|
| a | 기존 STO 토큰 수 |
| b | 기존 STO 토큰의 가격 (= 현재 NAV / a) |
| x | 신규 발행 토큰 수 |
| y | 신규 편입 PF의 **편입 시점 가치** (= 완공후가치 × 초기완공확률, P_COMPLETE_INIT=0.95 적용) |
| Q | 증자하고자 하는 자금조달액 (Q ≤ y) |
| P | 토큰 1개당 매입 비용 (공모가) |

**중요**: y는 완공 후 가치가 아니라 편입 시점의 실제 가치이다. 신규 PF는 아직 건설 중이므로 완공 확률(0.95)을 반영하여 할인한 가치를 사용한다. 이는 NAV 계산 방식과 일치하여 기존/신규 투자자 모두를 보호한다.

## 14.4 토큰 증자 조건 (3가지)

### 조건 1: 기존 주주 보호

```
b ≤ (y + ab) / (a + x)
```

새로운 가격은 기존 가격보다 높거나 같아야 함.

### 조건 2: 자금조달

```
P × x = Q
```

토큰 공모가 × 발행 토큰 수 = 자금조달액

### 조건 3: 신규 주주 보호

```
P ≤ (y + ab) / (a + x)
```

토큰 공모가보다 새로운 가격이 높거나 같아야 함.

## 14.5 낙찰 확률밀도함수

x_min과 x_max 범위 내에서 x_min 쪽에 편향된 지수분포로 샘플링.
토큰 적게 발행 → 기존투자자 보호(발행가 상승) 방향으로 편향.

```
p(t) ∝ exp(-bt),  b = 10.0,  t ∈ [0, 1]
x = x_min + t × (x_max - x_min)
```

# 15. 토큰 병합 (Reverse Split)

## 15.1 배경

롤오버 반복 시 토큰 수 누적 증가 + 기존 PF 만기로 NAV 감소 → 토큰 가격 하락 문제 해결.

## 15.2 설계

- **기준가**: Month 240 (20년차) 롤오버 직후의 토큰 가격
- **트리거**: 현재 토큰 가격 < 기준가 × 50%
- **병합 비율**: 2:1 (토큰 수 절반, 개당 가격 2배)
- **NAV 불변**: 펀드 총 가치는 변동 없음

## 15.3 피드백 효과

병합으로 b 상승 → x_max = y/b 감소 → 다음 롤오버에서 과도한 토큰 발행 억제
